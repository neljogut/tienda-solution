Necesito crear una plataforma web de gestión y venta para mi negocio.

La plataforma debe servir para vender dos tipos de productos:

1. Productos impresos en 3D.
2. Productos de reventa que no son impresiones 3D.

La app debe estar pensada para uso real de negocio, con catálogo, pedidos, clientes, inventario, cuenta corriente, caja, configuración de precios, dashboard, balance financiero, roles, permisos, imágenes en productos/insumos y generación de documentos PDF.

Debe tener una estética moderna, profesional, clara y fácil de usar. La interfaz debe ser responsive, usable en PC y celular, con navegación simple, tarjetas limpias, tablas claras, filtros, buscadores, estados visuales, imágenes bien presentadas y colores bien usados para diferenciar información importante.

Crear una plataforma profesional para administrar un negocio de impresión 3D y venta de productos, donde todo esté conectado:

* Catálogo.
* Pedidos.
* Clientes.
* Cuenta corriente.
* Inventario.
* Imágenes de productos, insumos y filamentos.
* Movimientos de inventario.
* Caja por turno.
* Pagos.
* Balance.
* Configuración de precios.
* Cotización automática del dólar.
* Roles y permisos.
* Documentos PDF para clientes y para control interno.

No quiero pantallas aisladas. Cada acción debe impactar donde corresponde.

# 3. Imágenes en productos, inventario e insumos

La plataforma debe permitir cargar imágenes en productos, filamentos, insumos y productos de reventa.

Esto es importante para que la app no se vea básica y para poder identificar visualmente cada elemento.

## Reglas generales de imágenes

Cada imagen debe poder:

* Subirse desde el dispositivo.
* Previsualizarse antes de guardar.
* Editarse o reemplazarse.
* Eliminarse.
* Tener una imagen principal.
* Tener imágenes adicionales si corresponde.
* Mostrarse correctamente en tarjetas, listados y detalles.
* Tener diseño responsive.
* Usar placeholder profesional si no hay imagen cargada.

El sistema debe validar:

* Formatos permitidos: JPG, JPEG, PNG, gif con movimiento, mp4.
* Tamaño máximo configurable.
* Que no se rompa la interfaz si una imagen no carga.
* Que las imágenes se vean bien escaladas, centradas y proporcionadas.

## Imágenes en productos del catálogo

Cada producto debe tener:

* Imagen principal obligatoria o recomendada.
* Galería opcional.
* Vista previa en formulario de creación/edición.
* Imagen visible en catálogo.
* Imagen visible en detalle del producto.
* Imagen visible en pedidos y PDFs si corresponde.

## Imágenes en filamentos

Cada filamento debe permitir imagen.

La imagen puede ser:

* Foto de la bobina.
* Foto del color.
* Muestra impresa.
* Imagen de referencia.
* Icono de bobina de filamento con el color de hex configurado

Además del código HEX del color, debe poder verse la imagen del filamento para identificarlo mejor.

En el listado de filamentos debe mostrarse:

* Miniatura de imagen.
* Marca.
* Material.
* Color.
* Código HEX.
* Peso disponible.
* Stock mínimo.

## Imágenes en insumos

Cada insumo debe permitir imagen.

Ejemplos:

* Imanes.
* Luces LED.
* Tornillos.
* Pegamento.
* Bolsas.
* Packaging.
* Etiquetas.
* Repuestos.

En el listado de insumos debe mostrarse:

* Miniatura de imagen.
* Nombre.
* Categoría.
* Stock actual.
* Stock mínimo.
* Costo unitario.

## Imágenes en productos de reventa

Cada producto de reventa debe permitir:

* Imagen principal.
* Galería opcional.
* Imagen visible en catálogo.
* Imagen visible en pedido.

## Uso de imágenes en PDFs

El PDF del cliente puede incluir imagen pequeña del producto si queda visualmente prolijo.

El PDF interno puede incluir imagen pequeña del producto para identificación, pero sin perjudicar la claridad del balance.

# 4. Precios del catálogo con cotización automática

Para productos impresos en 3D, el precio del filamento se carga en dólares dentro de la configuración de precios. (Configuracion de precios, aca colocamos la configuracion para que los pruductos que agregamos al catalogo se calcule automaticamente segun los parametros de precios, lo que se puede configurar en parametros de precios es : *Precio Filamento por KG (en dolares) *Precio KwH *Consumo (watts) *Vida util de la maquina *Costo en repuestos *Margen de error (%) *Multiplicador Minorista (productos que no son llavero) *Multiplicador minorista (productos que son llaveros) *% Descuento mayorista (Productos que no son llaveros) *% Descuento Mayorista (productos que son llaveros) *Umbral para aplicar descuento mayorista en gramos (productos que no son llaveros) *Umbral para aplicar descuento mayorista en gramos (productos que son llaveros) TODO ESO ES PARA PRODUCTOS DE IMPRESION 3D, AHORA FALTA LO QUE ES PARA PRODUCTOS DE REVENTA: *Ganancia sobre costo de compra % *Descuento mayorista % *Pedido minimo en $ para aplicar el descuento mayorista (0 queda deshabilitado el precio mayorista o colocar un switch que active o desactive el descuento mayorista en reventa)

El catálogo debe mostrar los precios en pesos argentinos.

La conversión debe hacerse automáticamente usando la cotización actual del dólar.

La cotización del dólar debe obtenerse automáticamente desde una API. Usar preferentemente una API confiable basada en la cotización del Banco Nación o una fuente equivalente para Argentina.

Reglas:

* El precio del filamento se guarda en dólares.
* El precio visible en el catálogo se muestra en pesos argentinos.
* El sistema debe consultar automáticamente la cotización del dólar.
* Si cambia la cotización del dólar, los precios del catálogo deben actualizarse automáticamente.
* No debo tener que editar producto por producto cuando cambia el dólar.
* Debe guardarse la última cotización obtenida.
* Debe guardarse fecha y hora de última actualización de la cotización.
* Si la API falla, el sistema debe usar la última cotización guardada.
* Si nunca hubo cotización guardada, permitir carga manual de emergencia solo para el dueño/admin.
* Mostrar en configuración cuál fue la última cotización usada.

La app debe dejar preparada una estructura clara para cambiar de proveedor de API en el futuro si fuera necesario.

# 5. Regla fundamental: precio congelado al momento de la venta

Los precios del catálogo pueden actualizarse automáticamente con la cotización del dólar, pero los pedidos ya creados no deben cambiar su precio.

Regla obligatoria:

* Cuando se crea un pedido, el sistema debe guardar una copia congelada del precio usado en ese momento.
* El pedido debe guardar:

  * Precio unitario vendido.
  * Precio mayorista o minorista aplicado.
  * Cotización del dólar usada en ese momento.
  * Fecha y hora de la cotización.
  * Costos calculados al momento de la venta.
  * Ganancia estimada al momento de la venta.
  * Imagen del producto o referencia visual si corresponde.
* Si mañana cambia el dólar, solo deben cambiar los precios del catálogo y productos todavía no vendidos.
* Los pedidos ya creados deben mantener el precio original.
* Cuenta corriente, pagos parciales, PDFs y balance deben respetar el precio congelado del pedido.
* Nunca recalcular un pedido viejo usando la cotización actual, salvo que el dueño edite manualmente el pedido y confirme la acción.

Esto es muy importante para que no se modifiquen deudas, pagos parciales, balances históricos ni facturas ya generadas.

# 6. Precio manual por producto

En cada producto del catálogo debe existir un switch llamado “Precio manual”.

Solo el dueño/admin puede activar o desactivar este switch.

Cuando el switch “Precio manual” está desactivado:

* El precio del producto se calcula automáticamente según los parámetros de precio.
* En productos impresos 3D se usa:

  * Costo de filamento.
  * Cotización del dólar.
  * Electricidad.
  * Mantenimiento.
  * Insumos.
  * Margen de error.
  * Multiplicadores.
  * Descuentos mayoristas.
* En productos de reventa se usa:

  * Costo de compra.
  * Porcentaje de ganancia.
  * Descuento mayorista si corresponde.

Cuando el switch “Precio manual” está activado:

* El dueño puede escribir manualmente el precio de venta minorista.
* Ese precio minorista manual reemplaza el precio minorista automático.
* El precio mayorista debe calcularse automáticamente en base al precio minorista manual y los parámetros de descuento mayorista configurados.
* El sistema debe seguir calculando costos internos y ganancia real, aunque el precio de venta sea manual.
* El producto debe mostrar un identificativo visual interno que diga “Precio manual”.
* Ese identificativo solo debe verlo el dueño/admin y los empleados con permiso.
* El cliente nunca debe ver que el producto tiene precio manual.
* En el listado interno de productos debe poder filtrarse por:

  * Precio automático.
  * Precio manual.

Cuando se crea un pedido con un producto de precio manual, el pedido debe guardar que ese producto fue vendido con precio manual.

# 10. Cuenta corriente

Crear una sección de cuenta corriente por cliente.

Esta sección debe mostrar todos los clientes que tienen deuda activa.

Por cada cliente debe verse:

* Nombre del cliente.
* Total adeudado.
* Cantidad de pedidos pendientes de pago.
* Fecha del pedido más antiguo pendiente.
* Botón para ver detalle.
* Botón para registrar pago.

Al entrar al detalle de un cliente debe mostrar:

* Todos los pedidos con saldo pendiente.
* Fecha de cada pedido.
* Total del pedido.
* Monto abonado.
* Saldo restante.
* Estado de pago.
* Historial de pagos realizados.

Desde cuenta corriente debo poder registrar un pago general del cliente.

Regla importante para aplicar pagos:

Cuando se registra un pago desde cuenta corriente, el dinero debe aplicarse automáticamente desde el pedido más antiguo hasta el más reciente.

Ejemplo:
Si un cliente debe $30.000 entre 3 pedidos:

* Pedido 1: debe $8.000.
* Pedido 2: debe $10.000.
* Pedido 3: debe $12.000.

Si el cliente paga $15.000:

* Se salda completo el Pedido 1 con $8.000.
* Los $7.000 restantes se aplican al Pedido 2.
* El Pedido 1 queda pagado y sale de cuenta corriente.
* El Pedido 2 queda señado o parcialmente abonado.
* El Pedido 3 queda pendiente.

Los pedidos saldados completamente deben desaparecer de la cuenta corriente activa, pero deben quedar en el historial.

Debe existir historial de pagos con:

* Fecha.
* Monto.
* Cliente.
* Pedidos afectados.
* Método de pago.
* Observación opcional.

# 18. Caja por turno

Agregar un módulo de caja.

La caja debe permitir abrir y cerrar caja por turno.

Solo el dueño o empleados con permiso pueden abrir y cerrar caja.

## Abrir caja

Al abrir caja se debe registrar:

* Fecha y hora de apertura.
* Usuario que abrió la caja.
* Monto inicial en efectivo.
* Observación opcional.
* Estado de caja: abierta.

No puede haber dos cajas abiertas al mismo tiempo para el mismo negocio, salvo que se implemente multi-sucursal en el futuro.

Si ya hay una caja abierta, el sistema debe avisar y no permitir abrir otra.

## Movimientos de caja

Durante el turno, la caja debe registrar automáticamente los movimientos relacionados a pagos.

Cada movimiento debe tener:

* Fecha y hora.
* Tipo de movimiento:

  * Ingreso por venta.
  * Pago de cuenta corriente.
  * Seña.
  * Egreso manual.
  * Ingreso manual.
  * Ajuste.
* Monto.
* Método de pago:

  * Efectivo.
  * Transferencia.
  * Mercado Pago.
  * Tarjeta.
  * Otro.
* Pedido asociado si corresponde.
* Cliente asociado si corresponde.
* Usuario que registró el movimiento.
* Observación opcional.

Cuando se registre un pago de pedido o cuenta corriente, debe impactar también en la caja abierta.

Si no hay caja abierta, el sistema debe mostrar una advertencia antes de registrar pagos presenciales.

## Cerrar caja

Al cerrar caja se debe mostrar un resumen del turno:

* Fecha y hora de apertura.
* Fecha y hora de cierre.
* Usuario que abrió.
* Usuario que cerró.
* Monto inicial.
* Total de ingresos.
* Total de egresos.
* Total esperado en caja.
* Total declarado al cierre.
* Diferencia de caja.
* Detalle por método de pago:

  * Efectivo.
  * Transferencia.
  * Mercado Pago.
  * Tarjeta.
  * Otro.
* Cantidad de pedidos cobrados.
* Pagos de cuenta corriente registrados.
* Observaciones de cierre.

El usuario debe ingresar el monto real contado al cierre.

El sistema debe calcular automáticamente si hay:

* Caja correcta.
* Sobrante.
* Faltante.

El cierre de caja debe quedar guardado en historial y no debe poder modificarse fácilmente. Si se permite corrección, debe quedar registrada como ajuste con motivo y usuario responsable.

# 20. Relación entre caja, pagos, pedidos y cuenta corriente

Reglas importantes:

* Todo pago registrado debe crear un movimiento de caja si hay caja abierta.
* Todo pago debe actualizar el pedido correspondiente.
* Todo pago parcial debe actualizar la cuenta corriente.
* Todo pago total debe quitar el pedido de cuenta corriente activa.
* Los pagos deben quedar en historial aunque el pedido ya esté pagado.
* La caja no calcula ganancias, solo movimientos de dinero.
* El balance financiero sí calcula ingresos, costos y ganancias reales.
* Caja y balance deben estar relacionados, pero no son lo mismo.

Diferencia importante:

* Caja muestra cuánto dinero entró y salió en un turno.
* Balance muestra rentabilidad, costos, ingresos y ganancias reales.
